# Echoes

GeoTagged Memories lets users capture, save, and relive moments with photos, descriptions, and map locations.

This README focuses on the Explore recommendation system used by the Explore page.

## Inspiration & What We Used

Primary GitHub inspiration/reference:

- https://github.com/huggingface/transformers.js

What we used from it in this project:

1. The `@huggingface/transformers` Node.js pipeline API for local inference.
2. `feature-extraction` pipeline for semantic text embeddings.
3. The `Xenova/all-MiniLM-L6-v2` embedding model for memory title + description vectorization.
4. Mean pooling + normalized embedding output pattern for similarity search readiness.

Project files where this is used:

- `server/utils/embeddingHelper.js`
- `server/routes/memoryRoutes.js`

## Explore Recommendation System

The Explore feed is generated by a hybrid engine that combines:

1. Semantic similarity (vector search on memory embeddings)
2. Geographic relevance (near recent interaction location)
3. Privacy and relationship filtering (public users + followed users)
4. Fallback recency feed when signals are weak

Main backend endpoint:

- GET /api/memory/explore

Client entry point:

- Explore page calls /api/memory/explore with optional category and searchQuery.

## Where It Lives

- Client Explore UI: client/src/pages/ExplorePage.jsx
- Explore API logic: server/routes/memoryRoutes.js (route: GET /explore)
- Embedding + clustering utils: server/utils/embeddingHelper.js
- Memory schema (embedding, geo index): server/models/memories.js

## Data Inputs Used For Recommendations

For a logged-in user, the backend builds context from:

1. Own memories (most recent first)
2. Liked memories (most recent first)
3. Optional active text query (searchQuery)
4. Optional category filter
5. Follow graph (to include followed private users)

## High-Level Flow

### A) Active Search Mode (searchQuery present)

1. Generate embedding from searchQuery.
2. Run MongoDB $vectorSearch on memories.embedding.
3. Apply filters:
	- Exclude current user memories
	- Apply category if provided
	- Keep results from public users OR followed users

### B) Passive Browsing Mode (no searchQuery)

1. Build interaction history = own memories + liked memories.
2. Deduplicate by memory id.
3. Extract available embeddings.
4. Stream A (Mood Market):
	- Cluster embeddings with k-means (k=3).
	- Pick cluster closest to most recent interaction embedding.
	- Use that centroid as target vector.
5. Stream B (Local Explorer):
	- Find most recent interaction with location.
	- Run geo query around that location ($geoNear, 50km).

### C) Blend Streams

1. Interleave semantic and geo results.
2. Deduplicate by memory id.
3. Return merged feed.

### D) Fallback Stream

If hybrid result is empty:

1. Fetch recent public posts (excluding current user).
2. Return latest items by createdAt descending.

## Embedding Lifecycle

Embeddings are generated asynchronously to keep write latency low:

1. On memory create:
	- API responds immediately after save.
	- Embedding generation runs in background.
2. On memory edit (title/description changed):
	- API responds immediately.
	- Embedding regeneration runs in background.

Embedding model:

- Xenova/all-MiniLM-L6-v2 via @huggingface/transformers

Notes:

- Vectors are mean-pooled and normalized.
- Embedding field is hidden by default in schema (select: false).

## MongoDB Requirements

The Explore engine depends on two index capabilities:

1. Vector index for embedding search
	- Used by $vectorSearch
	- Route expects index name: vector_index
2. Geospatial index for location search
	- 2dsphere index exists on memory location field in schema

If vector index is missing or unsupported, semantic stream will fail.

## Current Ranking Logic Summary

The ranking is currently heuristic, not a weighted scoring model:

1. Semantic list is produced
2. Geo list is produced
3. Lists are interleaved one-by-one
4. Duplicates are removed

This gives diversity between concept relevance and location relevance.

## Client Behavior (Explore Page)

Explore page sends:

- category: selected category chip
- searchQuery: text query from search bar

Then renders returned memories in a grid and opens PostModal on click.

## Privacy + Access Rules in Feed

A memory can appear if:

1. It does not belong to current user.
2. Owner is public OR current user follows them.

This rule is applied after joining memory with user document.

## Quick Test Checklist

1. Create a few memories with varied text and categories.
2. Wait for background embeddings to complete.
3. Open Explore with no query:
	- Verify recommendations are returned.
4. Use search query (for example: beach sunset):
	- Verify semantic shift in results.
5. Set category filter:
	- Verify feed narrows correctly.
6. Check private user visibility:
	- Followed private user should appear.
	- Unfollowed private user should not appear.

## Known Constraints

1. Cold start: if user has no interaction embeddings, system may rely more on fallback.
2. Background embedding delay: newly created/edited memories may take a short time to become semantically searchable.
3. Randomness in k-means init can slightly vary passive mood centroids.

## Suggested Next Improvements

1. Replace interleaving with explicit scoring and weighted blending.
2. Add interaction recency decay weighting.
3. Cache per-user target vectors for short intervals.
4. Add metrics logging (CTR, dwell, save, like) for online evaluation.
5. Backfill embeddings for old memories via batch job.
